============================================================
  Graph RAG -- Setup Documentation
  What was built, how, and why
============================================================

This document explains every step taken to set up the Graph RAG
system on this machine. Written for future reference or if you
ever need to redo this on a new machine.


------------------------------------------------------------
WHAT IS GRAPH RAG?
------------------------------------------------------------

RAG stands for Retrieval-Augmented Generation. Instead of
asking an AI a question from memory alone, RAG first searches
a local knowledge base of your own documents, pulls out the
most relevant pieces, and feeds them to the AI as context.
The AI then answers using that context -- grounded in your data.

"Graph" RAG goes a step further: it also builds a knowledge
graph (a network of entities and relationships extracted from
your documents). When you ask a question, it retrieves chunks
not just by text similarity, but also by following connections
in the graph -- finding related content you might have missed.

This setup works fully offline using LM Studio, or online
using the Claude API. You can switch between them anytime.


------------------------------------------------------------
PART 1 -- THE SUPABASE DATABASE (Docker)
------------------------------------------------------------

WHAT:
  Supabase is an open-source backend platform built on top of
  PostgreSQL. We use it for its database, REST API, and web
  dashboard. The key feature we need is pgvector -- a PostgreSQL
  extension that lets us store and search embeddings (the
  numerical representations of text used for similarity search).

SERVICES:
  - graph_rag_db     : PostgreSQL 15 with pgvector pre-installed
  - graph_rag_rest   : PostgREST (turns the DB into a REST API)
  - graph_rag_meta   : Postgres Meta (used by the Studio UI)
  - graph_rag_kong   : Kong API gateway (routes requests for Studio)
  - graph_rag_studio : Supabase Studio (web dashboard at localhost:3001)

WHAT THE DATABASE CONTAINS:
  - documents    : The full text of every ingested document
  - chunks       : Documents split into smaller searchable pieces
  - embeddings   : Vector representations of each chunk (384 numbers)
  - graph_nodes  : Entities extracted from text (people, places, concepts)
  - graph_edges  : Relationships between those entities
  - match_chunks(): A SQL function that finds similar chunks by vector

PART 3 -- PYTHON DEPENDENCIES
------------------------------------------------------------

  Database:
    psycopg2-binary  Connects Python to PostgreSQL
    pgvector         Lets psycopg2 read/write vector columns

  Embeddings:
    sentence-transformers  The embedding library
    torch                  Required by sentence-transformers
    Model used: all-MiniLM-L6-v2 (384 dimensions, fast, accurate)
    After the first run, the model is cached and works offline.

  NLP and Graph:
    spacy      Extracts named entities from text
    networkx   In-memory graph data structure
    nltk       Sentence tokenizer

  LLM Clients:
    openai     Used to talk to LM Studio (OpenAI-compatible)
    anthropic  Used to talk to the Claude API

  File Support:
    pypdf      Reads PDF files

  Utilities:
    python-dotenv  Reads the .env file
    PyJWT          Generates JWT tokens
    numpy          Array math


------------------------------------------------------------
PART 4 -- LLM PROVIDERS
------------------------------------------------------------

  LMSTUDIO (default, fully offline):
    - Requires LM Studio app to be running
    - Enable Local Server in LM Studio (Settings -> Local Server)
    - Load any model in LM Studio
    - Connects to http://localhost:1234/v1

  CLAUDE (Anthropic API, requires internet):
    - Requires an API key from https://console.anthropic.com/
    - Set ANTHROPIC_API_KEY in .env
    - Default model: claude-sonnet-4-6


------------------------------------------------------------
PART 5 -- HOW A QUERY WORKS (end to end)
------------------------------------------------------------

  1. Your question is encoded into a vector (384 numbers)
     using the all-MiniLM-L6-v2 embedding model locally.

  2. That vector is sent to PostgreSQL via match_chunks(),
     which uses pgvector to find the most similar chunks.

  3. The chunk IDs from step 2 are used to look up graph nodes.

  4. The graph is traversed 1-2 hops out from those nodes,
     collecting IDs of additional related chunks.

  5. All collected chunks are fetched and assembled into context.

  6. Context is inserted into a prompt and sent to the LLM.

  7. The response is returned to the terminal.


------------------------------------------------------------
PART 6 -- ADDING YOUR OWN DOCUMENTS
------------------------------------------------------------

  In the terminal:
    py main.py ingest path/to/yourfile.txt
    py main.py ingest path/to/yourfile.pdf

  While chat is running:
    /ingest path/to/yourfile.txt

  Supported formats: .txt  .md  .pdf

  After ingesting several documents, rebuild the graph:
    py main.py build-graph


------------------------------------------------------------
PART 7 -- IF YOU NEED TO START OVER
------------------------------------------------------------

  1. Stop and remove containers + volumes:
       docker compose down -v

  2. Start fresh:
       docker compose up -d

  3. Wait ~15 seconds, then run the init SQL:
       docker exec -i graph_rag_db psql -U postgres < sql/init.sql

  4. Regenerate .env:
       py generate_keys.py
